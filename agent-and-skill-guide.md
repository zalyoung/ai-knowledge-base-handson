# Agent 与 Skill 使用最佳实践

## 核心理念

**Agent = 决策者，Skill = 执行者**

Agent 负责"谁来做、能做什么、底线是什么"，Skill 负责"什么时候用、怎么做、输出什么"。这种分离最大化激发 Agent 的自主性——Agent 根据任务选择合适的 Skill，而不是被预设规则束缚。

## 职责边界

```
┌─────────────────────────────────────────────────────────────┐
│                    Agent（决策层）                            │
├─────────────────────────────────────────────────────────────┤
│  • 角色定位：我是谁、负责什么                                  │
│  • 权限控制：能用什么工具、禁止什么                             │
│  • 质量红线：必须遵守的底线规则                                │
│  • 推荐 Skill：列出可搭配的 Skill（仅供参考）                  │
│  ❌ 不包含：具体业务规则                                       │
└─────────────────────────────────────────────────────────────┘
                              │
                              ▼
┌─────────────────────────────────────────────────────────────┐
│                    Skill（执行层）                            │
├─────────────────────────────────────────────────────────────┤
│  • 使用场景：什么时候调用                                      │
│  • 执行步骤：具体怎么做                                       │
│  • 业务规则：评分标准、字数、数量等                             │
│  • 输出格式：JSON 结构定义                                    │
│  ❌ 不包含：权限控制、质量红线                                 │
└─────────────────────────────────────────────────────────────┘
```

## 调用方式

**必须显式指定 Skill**，明确告诉 Agent 使用哪个技能：

```
✅ 正确："使用 github-trending 技能采集本周热门项目"
✅ 正确："调用 tech-summary 技能分析最新采集数据"
❌ 错误："采集一下热门项目"（Agent 不知道用哪个 Skill）
```

## 冲突解决

当 Agent 定义与 Skill 定义出现冲突时：

1. **Skill 中的业务规则优先**——这是调用时指定的，具有最高优先级
2. **Agent 中的质量红线始终生效**——这是底线，不可被 Skill 覆盖

示例：
- Agent 说"禁止编造"，Skill 说"允许推断" → 仍禁止编造（红线优先）
- Agent 说"摘要 100-300 字"，Skill 说"摘要 ≤50 字" → 按 Skill 的 ≤50 字（业务规则优先）

## 常见错误

### 错误 1：Agent 中定义业务规则

```markdown
# ❌ 错误示例（collector.md）
## 工作流程
### 第一步：搜索热门仓库
- 查询参数：q=stars:>1000...
- 获取最近一周内活跃项目
```

**问题**：业务规则应该放在 Skill 中，Agent 只负责权限和红线。

### 错误 2：Skill 中定义权限控制

```markdown
# ❌ 错误示例（github-trending/SKILL.md）
## 权限说明
| 工具 | 许可 |
|------|------|
| Write | 禁止 |
```

**问题**：权限控制应该放在 Agent 中，Skill 只负责执行逻辑。

### 错误 3：调用时不指定 Skill

```
用户："帮我分析一下数据"
Agent：不知道用什么规则，可能用默认规则或报错
```

**问题**：没有显式指定 Skill，执行结果不可预测。

## 创建新 Skill 的检查清单

创建 Skill 时，确保包含以下内容：

- [ ] YAML frontmatter（name、description、allowed-tools）
- [ ] 使用场景（什么时候调用这个 Skill）
- [ ] 执行步骤（Step 1, 2, 3...）
- [ ] 业务规则（评分标准、字数、数量等具体参数）
- [ ] 输出格式（JSON Schema）
- [ ] 约束（业务相关限制，如评分分布）

**不要包含**：
- 权限控制（这是 Agent 的职责）
- 质量红线（这是 Agent 的职责）

## 目录结构

```
.opencode/
├── agents/                      # Agent 定义（决策层）
│   ├── collector.md             # ~50行：角色 + 权限 + 红线
│   ├── analyzer.md              # ~50行：角色 + 权限 + 红线
│   └── publisher.md             # ~50行：角色 + 权限 + 红线
│
└── skills/                      # Skill 定义（执行层）
    ├── github-trending/         # GitHub 采集技能
    │   └── SKILL.md             # ~150行：步骤 + 规则 + 格式
    ├── tech-summary/            # 技术分析技能
    │   └── SKILL.md             # ~150行：步骤 + 规则 + 格式
    └── ...
```

## 一句话总结

> **Agent 决定"做什么"，Skill 决定"怎么做"。调用时必须显式指定 Skill，让 Agent 自主选择执行策略。**
